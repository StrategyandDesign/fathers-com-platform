# Fathers.com Platform Architecture

Status: Approved baseline. Changes to this document go through PR review.
Companion documents: CONTRIBUTING.md (process), FC-REQ-001 (Phase 1 requirements), supabase/functions/esign-bridge/ESIGN_INTEGRATION.md (e-sign boundary).

## 1. The architecture in one paragraph

Fathers.com is a generated static front-end served by Vercel, backed by Supabase (Postgres, Auth, Storage) where row-level security is the enforcement boundary, with Supabase Edge Functions as the only home for secrets and money. Pages are generated from Python builders (`build_pages.py`, `build_dashboards.py`) and committed; the deploy is a git push. This is deliberate: minimal attack surface, near-zero hosting cost for a nonprofit, instant rollback (static files), and security enforced in the database rather than in browser code.

## 2. Principles (what a reviewer should hold us to)

1. **RLS is the boundary.** The browser is untrusted. Every table carries row-level security; a father reaches only his own rows; admins are gated by the `user_roles` table. Client-side checks are UX, never security.
2. **Secrets and money live server-side only.** The Supabase anon key in `assets/js/config.js` is public by design. Service-role keys, e-sign tokens, and payment keys exist only as edge-function secrets (`supabase secrets set`). Price and eligibility are decided server-side: the checkout function verifies the participant claim and reads price from the database before fulfillment. The browser computes neither.
3. **Generated, never hand-edited.** All HTML is generated from the builders. CI fails any push where regenerating produces a diff.
4. **Scoring is versioned configuration plus a pure function.** Per FC-REQ-001 Appendix C: the Keystone item bank and scoring model are versioned data; scoring is deterministic; every stored result records the versions that produced it. Golden fixtures verify scoring in CI as they are introduced.
5. **Privacy by construction.** No PII in URLs. Voice recordings live in a private bucket under per-user folders. Consent precedes collection. A father's data is his to delete. Admins see aggregates, not individuals (FR-V1-023).
6. **Ship dark, flip flags.** New surfaces land behind a flag (a `flags` table read at runtime) so rollout and rollback are data changes, not deploys.

## 3. System map

- **Front-end**: static HTML/CSS/JS, FORGE design system, generated by Python builders, hosted on Vercel. Every branch gets an automatic Vercel preview deploy.
- **Auth**: Supabase Auth, email and password only. Account creation, sign-in, and reset all use the password flow; the magic link is retired platform-wide. `?next=` redirects are sanitized against an allowlist pattern.
- **Data**: Supabase Postgres. Schema history lives in `supabase/migrations/` (see §5). RLS on every table.
- **Storage**: private `voice` bucket, per-user folder policies.
- **Edge functions**: `checkout` (the sole enrollment path: verifies the participant claim, reads price from the database, fulfills at zero dollars; priced rows route to Stripe when enabled), `checkout-webhook` and `stripe-webhook` (signature-verified Stripe fulfillment), `issue-certificate`, `esign-bridge` (holds the e-sign API token; re-authorizes server-side against `certificate_awards`), and `send-email` (Resend).
- **CI**: GitHub Actions (`.github/workflows/ci.yml`): config integrity, build determinism, JS syntax, offline e2e smoke tests.

## 4. Environments

- **Production**: Vercel production deployment from `main`, Supabase project `kemqpiboqeqhbuuldmls`.
- **Previews**: automatic per-PR Vercel deployments for visual review. Known limitation: previews currently read the production `config.js`, so previews are for review, not destructive testing; destructive testing happens in the offline test suite.
- **Staging (planned)**: a second Supabase project plus a build-time step that writes `config.js` from environment variables per environment. Until then, database changes are rehearsed by applying migrations to a scratch project via the Supabase CLI before production.

## 5. Database change management

All schema changes are files under `supabase/migrations/`, applied with the Supabase CLI (`supabase db push`). Hand-running SQL in the dashboard editor without a corresponding migration file is prohibited after the baseline is taken (adoption procedure in `supabase/migrations/README.md`). Historical hand-applied SQL is archived in `supabase/sql-archive/` for the record.

## 6. Acknowledged technical debt (tracked, not hidden)

1. **CSP allows `unsafe-inline` and `unsafe-eval`** (vercel.json) pending a refactor of inline handlers. Tightening is planned; RLS and edge-function authorization are the security boundary in the meantime.
2. **Previews point at production config** (§4). Mitigated by offline tests; resolved by the staging config step.
3. **No error telemetry.** Sentry (or equivalent) to be added with a DSN; until then, errors surface in admin consoles loudly by design (see admin-certs.js pattern).
4. **Supabase built-in email sender is capped (2/hour)**; production sign-in email requires custom SMTP (Resend) before any launch push.

## 7. Decision records

- **ADR-1: Keep the static+Supabase architecture; harden process.** Rebuilding on a server framework adds cost and attack surface without serving Phase 1. Revisit when server-rendered personalization is required at scale.
- **ADR-2: Edge functions are the only server.** Anything touching secrets, money, signatures, or admin-grade writes that bypass RLS uses an edge function with service-role access and explicit authorization checks (the esign-bridge pattern).
- **ADR-3: Enrollment and payments through one server-side function.** The browser never computes a price and never decides eligibility. The checkout function verifies the participant claim, reads price from the database, and fulfills; priced rows (the facilitator course, later) create a Stripe session and the signature-verified webhook fulfills through the same code path.
- **ADR-4: Feature flags as data, build flags for whole surfaces.** A `flags` table read at runtime governs in-page rollout. Whole surfaces ship dark behind build flags: `SHOW_MILITARY` in `build_pages.py` gates the veteran pages and their assets, per POSITIONING.md.
- **ADR-5: Claims gate the courses.** A participant enrolls only when a Certified Facilitator or Certified Organization has claimed him (`participant_claims`, RLS-scoped). The free Profile, plan, and film library remain open; the courses and the Certificate of Completion flow through claims.
